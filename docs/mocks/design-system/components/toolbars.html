Toolbars & chrome

Two pieces of fixed chrome frame every Trellis screen: the 35px title bar across the top (VS Code-style: command center, layout toggles, window controls) and the 48px vertical activity rail on the left, topped by the hamburger application menu (vscode.dev-style).

Title bar

Derived from: features/editor/title-bar/ — .tds-title-bar.

Left: an empty spacer (the app menus live behind the rail's hamburger) that keeps the command center centered. Center: the command-center pill holding the open document's name. Right: layout toggles (primary-sidebar toggle shown active with its filled band) and window controls.

PartSpec
Bar35px, chrome surface, bottom hairline, 13px text, three flex regions (1 / 1.5 / 1)
Left regionEmpty spacer; keeps its flex share so the command center stays centered — the app menus are the rail hamburger's (see Menus)
Command center24px pill, max-width 38rem, raised surface + strong border, 12px muted text; hover lightens to overlay + primary ink; title is "«document» — Trellis"
Layout toggles26×24, 14px pane glyphs (stroke 1.2); active pane band fills at 35% via .tds-pane-fill; aria-pressed
Window controls46px wide, full height, 11px glyphs (stroke 1); close hovers --tds-window-close #c42b1c with white ink — the Windows convention, always last

Activity rail

Derived from: features/editor/editor-toolbar/ — .tds-rail. Buttons documented in Buttons; connection dot in Indicators.

Connected

Canonical order: Application Menu (the hamburger, always first), Explorer (only in browsers with the File System Access API), Templates, Documents, Explain This. The New/Save/Upload actions live in the hamburger's File submenu (see Menus). The connection dot pins to the bottom via the bottom group's margin-top: auto. Documents is shown active.

PartSpec
Rail48px wide, chrome surface, right hairline, 0.5rem gaps and vertical padding
GroupsTop: hamburger, then panel toggles in canonical order. Bottom: status, pinned by margin-top: auto
HamburgerA menu-trigger rail button: aria-haspopup="menu" + aria-expanded; stays filled while its menu is open; flyout documented in Menus
Feature gatingButtons for unavailable capabilities are removed, not disabled (Explorer without FS Access API)

Rules

Do Keep the close button rightmost with its red hover — users aim for it from muscle memory.
Do Reflect pane visibility in the layout toggles' filled bands; the toggle is the layout's single source of truth in the chrome.
Don't Add text, badges, or notification dots to the rail — it is icons plus one status dot, nothing else.
Don't Reorder rail actions per screen; the canonical order is fixed.
