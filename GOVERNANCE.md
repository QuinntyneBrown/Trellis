# Governance

Trellis is governed as a maintainer-led open-source project. The governance model is intentionally lightweight while the project is early in its lifecycle.

## Roles

### Maintainers

Maintainers are responsible for project direction, release decisions, review standards, security response, dependency policy, and repository administration.

### Contributors

Contributors improve the project through issues, documentation, design feedback, tests, code, and reviews. Regular, high-quality participation may lead to maintainer nomination.

## Decision Making

Most decisions are made through pull request review and issue discussion. Maintainers seek rough consensus, but they may make a final decision when a decision is needed to keep the project moving or to protect quality, security, or licensing posture.

Large changes should be proposed in an issue before implementation. Examples include architectural changes, storage migrations, public API changes, dependency replacements, and major user experience changes.

## Releases

Until the first stable release, `main` represents the current development line. Release notes are tracked in [CHANGELOG.md](CHANGELOG.md). Stable release processes will be documented before the project publishes a `1.0.0` release.

## Maintainer Changes

Maintainers may nominate additional maintainers based on sustained contribution quality, sound judgment, and constructive project participation. Maintainer removals should be rare and based on inactivity, security concerns, repeated conduct violations, or loss of project trust.
