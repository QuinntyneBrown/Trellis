# Security Policy

The Trellis maintainers take security reports seriously.

## Supported Versions

Until Trellis publishes tagged stable releases, security fixes are made on `main`. After the first stable release, this file will be updated with supported release lines.

## Reporting a Vulnerability

Do not disclose suspected vulnerabilities in public issues, pull requests, discussions, or social channels.

Use GitHub private vulnerability reporting if it is enabled for this repository. If private vulnerability reporting is not available, open a minimal public issue asking for a maintainer contact channel and omit technical details until a private channel is established.

Include the following when possible:

- Affected commit, branch, or version.
- Steps to reproduce.
- Expected and actual impact.
- Relevant logs, screenshots, or proof-of-concept code.
- Whether the issue is already publicly known.

## Response Expectations

Maintainers should acknowledge a valid private report within five business days, triage severity, and coordinate disclosure timing with the reporter. Fixes may be released as commits to `main`, release branches, or tagged releases depending on project maturity at the time.

## Scope

Security reports should focus on vulnerabilities in Trellis source code, configuration, packaging, or documented deployment guidance. Vulnerabilities in third-party dependencies should also be reported to the affected upstream project.
